# Implementation Plans — sudoprint

Generated by the improve skill on 2026-06-13. This is a **greenfield build**: the
plans were written against commit `e90880a`, when the repo contained only
`PLAN.md` (the design spec), this `plans/` set, and the git scaffold
(`.gitignore`, `.gitattributes`). Each plan's drift check compares against that
SHA. These plans decompose `PLAN.md` into ordered, independently-verifiable
steps for an executor. Execute in numeric order — each plan depends on the ones
before it. Each executor: read the plan fully before starting, honor its STOP
conditions, run every verification command, and update your row below when done.

Three design decisions were made during the review and are baked into the plans:

1. **Text rendering uses `golang.org/x/image/font/opentype`**, not
   `github.com/golang/freetype` (the latter is frozen/unmaintained).
2. **Difficulty clue counts are a target within an accepted range, and the actual
   count is reported** per puzzle (random removal cannot always hit the exact
   target, especially for `hard`).
3. **Grid variety** comes from band/stack/row/column shuffles + relabel +
   optional transpose on the base solved grid, not a single fixed pattern.

## Dependency notes

- 002 before 003: generation proves uniqueness via the solver's `CountSolutions`.
- 003 before 004 and 006: rendering and the CLI need the `puzzle.Puzzle` type.
- 004 before 005: the PDF bundles `image.Image` values produced by `RenderPage`.
- 006 last (of the core build): it wires generator + image + PDF together and is
  where the end-to-end smoke test lives.
- 007 (manifest) depends on 006 — it records per-puzzle data produced in `run`.
- 008 (symmetric removal) depends on 003 — it reopens `puzzle/generate.go`'s
  `carve`. 007 and 008 do not depend on each other.

The verification gate is identical across all plans: `go build ./...`,
`go vet ./...`, `go test ./...` must each exit 0. Plan 001 establishes it.

## What was reviewed (and what wasn't)

Reviewed: the `PLAN.md` design spec end-to-end — generation algorithm, solver
use, rendering geometry, library choices, PDF unit mapping, CLI behavior. No
source code was audited because none exists yet.

Findings folded into the plans (with the plan that fixes each):

- No test strategy at all → tests are first-class in every plan (001 baseline,
  002/003/004/005/006 each ship tests).
- Difficulty targets not guaranteed → 003 (range + report actual clue count).
- Limited grid variety → 003 (full shuffle suite).
- `freetype` is legacy → 004 (`x/image/font/opentype`).
- PDF image/DPI/points mapping + gopdf API risk → 005 (explicit point rect +
  API verification step with fallback).
- Cell-size rounding drift → 004 (rounded line positions).
- Gridline draw order → 004 (thin-then-thick draw order specified).
- Output overwrite behavior undefined → 006 (`os.Create` overwrite, documented).
- `go.mod` module path / Go version not pinned → 001 (`sudoprint`, `go 1.22`).

## Still deferred (not planned yet)

- **Batch-generation concurrency** — parallelize generation across cores for
  large `-n`. Only worth it at scale, and it changes `-seed` semantics
  (per-puzzle derived seeds), so it must land as one contained, well-tested
  change. Not planned now.
- **Streaming pages to the PDF** instead of holding all images in memory —
  only matters at large `-n`; noted in 006's maintenance notes.
- **`-symmetric=false` escape** for minimal-clue asymmetric puzzles — would
  touch `Generate`'s signature; noted in 008's maintenance notes.
